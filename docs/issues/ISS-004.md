# explicit-module-boundary-types et explicit-function-return-type : diagnostics en double

**Règles** : `ts-explicit-function-return-type`, `ts-explicit-module-boundary-types`
**Impact** : 168 diagnostics redondants sur Amadeo (sur les return types des fonctions exportées)
**Priorité** : moyenne

## Problème

Les deux règles émettent un diagnostic sur le return type manquant des fonctions exportées :
- `ts-explicit-function-return-type` cible **toutes** les fonctions (exported ou non)
- `ts-explicit-module-boundary-types` cible uniquement les fonctions **exportées** mais vérifie aussi les **types de paramètres**

Sur une fonction exportée sans return type, les deux règles émettent un warning — doublon pur sur cet axe.

### Différences réelles

| Aspect | `function-return-type` | `module-boundary-types` |
|---|---|---|
| Scope | Toutes les fonctions | Seulement les exports |
| Return type | Oui | Oui |
| Types de paramètres | Non | Oui |
| Méthodes de classe | Oui | Non |

Les deux sont enregistrées via `register_ts_family!` sur les mêmes langages.

## Fix proposé

**Option A — suppression mutuelle :** dans `ts-explicit-function-return-type`, skip les fonctions exportées si `ts-explicit-module-boundary-types` est activée (ou l'inverse). Nécessite que le check ait accès à la config globale des règles actives.

**Option B — documentation + defaults :** `ts-explicit-function-return-type` est déjà `disabled = true` dans `defaults.toml:156`. Documenter clairement que les deux ne doivent pas être activées simultanément. Mais ça n'empêche pas le cas d'Amadeo.

**Option C — fusion :** merger les deux en une seule règle avec un mode `all` vs `exports-only` configurable. Plus propre mais breaking change.

## Fichiers concernés

- `src/rules/ts_explicit_function_return_type/typescript.rs`
- `src/rules/ts_explicit_module_boundary_types/typescript.rs`
- `src/config/defaults.toml:156` — `ts-explicit-function-return-type` déjà disabled
