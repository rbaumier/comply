# no-hardcoded-secret : faux positifs sur JSX et formulaires

**Règle** : `no-hardcoded-secret`
**Impact** : 5 faux positifs sur Amadeo, 3 FP sur hook0
**Priorité** : moyenne
**Statut** : partiellement résolu (633cf7e0) — doc comments (`///`, `//!`, `/** */`) exemptés, test defaults (`postgres:postgres@localhost`) exemptés. Reste à faire : exclusion JSX attributes et i18n (Amadeo-spécifique).

## Problème

L'heuristique flagge des lignes de JSX qui contiennent le **mot** "key" ou "password" sans qu'il y ait de secret hardcodé :

1. **`api-keys.lazy.tsx` l.86-87** — template i18n contenant le mot "key" dans une chaîne de traduction (`t("api_keys.description")`), pas un vrai secret.
2. **`login.lazy.tsx` l.75/80** — champs de formulaire avec `name="password"` ou `type="password"`. Ce sont des éléments d'UI, pas des credentials hardcodées.

## Analyse

L'heuristique semble chercher des patterns comme `variable_name = "valeur"` où le nom contient un mot sensible (`key`, `secret`, `password`, `token`). Mais elle matche aussi :
- Des attributs JSX (`name="password"`)
- Des clés i18n qui contiennent le mot dans le namespace
- Des labels de formulaire

## Fix proposé

Affiner l'heuristique :
1. **Exclure les attributs JSX** — un `<input type="password">` n'est pas un secret hardcodé. Si le nœud parent est un `jsx_attribute`, skip.
2. **Exclure les appels i18n** — si la string est dans un appel `t()`, `i18n.t()`, `useTranslation`, skip.
3. **Exiger une affectation** — ne flagguer que quand il y a un pattern `const/let/var <name_sensible> = "<valeur_littérale>"` ou `<name_sensible>: "<valeur>"` dans un objet, pas la simple présence du mot dans du texte.
