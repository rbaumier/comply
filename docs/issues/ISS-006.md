# no-timing-attack : faux positif sur validation de formulaire client-side

**Règle** : `no-timing-attack`
**Impact** : 1 faux positif sur Amadeo, 3 FP sur hook0
**Priorité** : basse (mais impacte la crédibilité de la règle)
**Statut** : ✅ résolu (633cf7e0) — comparaisons contre littéraux (`null`, `undefined`, `""`, `false`, nombres) exemptées + exemption confirmation-pattern existante couvre Zod `.refine()`

## Problème

La règle voit `password === confirmPassword` dans un schéma de validation Zod et suppose que c'est une vérification de secret côté serveur nécessitant `crypto.timingSafeEqual`.

En réalité, c'est une validation de formulaire "confirmer le mot de passe" — l'utilisateur compare ses propres inputs. Même si exécuté côté serveur en validation d'input, il n'y a aucun vecteur d'attaque timing (l'attaquant contrôle les deux valeurs).

## Contexte

Une timing attack sur `===` nécessite :
1. Que l'une des valeurs soit un **secret** stocké (hash, token)
2. Que l'attaquant puisse mesurer le temps de réponse
3. Que la comparaison court-circuite sur le premier byte différent

Ici aucune des deux valeurs n'est un secret — ce sont deux champs de formulaire.

## Fix proposé

Exclure les cas où :
1. La comparaison est dans un **schéma de validation** (Zod `.refine()`, `.superRefine()`, Yup `.test()`, Valibot)
2. Les deux opérandes viennent du même objet de données (ex: `data.password === data.confirmPassword`)
3. Le fichier est un fichier de validation/schema (heuristique de nom : `*schema*`, `*validation*`, `*form*`)

La détection la plus robuste serait (1) : si le `===` est dans un callback passé à `.refine()` ou `.superRefine()`, le contexte est clairement de la validation, pas de l'authentification.
