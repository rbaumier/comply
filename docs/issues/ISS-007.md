# no-generic-names : trop agressif sur les patterns idiomatiques

**Règle** : `no-generic-names`
**Impact** : 75 faux positifs sur Amadeo
**Priorité** : moyenne
**Statut** : partiellement résolu (merge worktree-fp-hunting) — guard test/storybook ajouté, patterns idiomatiques toujours ouverts

## Problème

La règle interdit des noms comme `data`, `value`, `result`, `item` partout. Mais dans de nombreux contextes ces noms sont parfaitement clairs et idiomatiques :

1. **Callbacks de `.map()` / `.filter()` / `.forEach()`** : `items.map(item => ...)` — `item` est le nom idiomatique quand le type est évident du contexte.
2. **Destructuring de hooks** : `const { data } = useQuery(...)` — `data` est le nom exposé par l'API du framework, le renommer serait artificiel.
3. **Destructuring de réponses** : `const { data } = await fetch(...)` — même chose.
4. **Variables de scope courte** : dans un bloc de 3 lignes, `result` est parfaitement clair.

## Fix proposé

Ajouter des exceptions contextuelles :
1. **Paramètre de callback inline** — si le nom est le paramètre d'une arrow function passée directement à `.map()`, `.filter()`, `.find()`, `.reduce()`, `.forEach()`, `.some()`, `.every()`, ne pas flagguer.
2. **Destructuring de retour de fonction** — si le nom vient d'un destructuring (`const { data } = ...`), l'utilisateur ne choisit pas le nom, c'est l'API qui l'impose.
3. **Scope courte** — si la variable n'est utilisée que dans les N lignes suivantes (ex: N=5), le nom générique est acceptable.

L'option (2) est la plus impactante et la plus simple à implémenter : un `property_identifier` dans un `object_pattern` d'un `variable_declarator` ne devrait pas être flaggué.
