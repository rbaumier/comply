# react-jsx-no-jsx-as-prop : faux positifs sur les patterns shadcn/Radix

**Règle** : `react-jsx-no-jsx-as-prop`
**Impact** : 23 faux positifs sur Amadeo
**Priorité** : basse

## Problème

La règle interdit de passer du JSX en prop (`icon={<Icon />}`, `trigger={<Button />}`). Le rationnel est d'éviter des re-renders inutiles (le JSX crée un nouvel objet à chaque render).

Mais ce pattern est le standard de shadcn/ui et Radix UI :
- `<Dialog trigger={<Button>Open</Button>}>`
- `<DropdownMenu trigger={<IconButton />}>`
- `<Tooltip content={<span>Help</span>}>`

Ces composants sont conçus pour recevoir du JSX en prop — c'est leur API.

## Fix proposé

**Option A — whitelist de props :** ne pas flagguer les props connues des composants Radix/shadcn (`trigger`, `content`, `icon`, `asChild`, `overlay`, etc.).

**Option B — détection framework :** si shadcn/Radix est détecté (présence de `components.json` ou `@radix-ui/*` dans les deps), relâcher la sévérité ou désactiver.

**Option C — scope au composant :** ne flagguer que si le composant parent est un composant custom du projet, pas un composant de bibliothèque. Difficile à implémenter sans résolution de types.

L'option A est la plus pragmatique.

## Également mentionné

- **`react-require-content-visibility`** (8 FP) : exige `contentVisibility: auto` ou un virtualizer sur chaque `.map()` en JSX. La plupart des listes sont petites (< 50 items). Considérer un seuil configurable ou désactiver par défaut.
- **`symmetric-pairs`** (19 FP) : impose d'avoir un create/delete, open/close pour chaque pair. Trop rigide — certaines ressources n'ont pas de symétrie (ex: un composant `OpenDialog` sans `CloseDialog`).
- **`no-multi-op-oneliner`** (23 FP) : flagge les chaînes Playwright `page.getByRole(...).fill(...).click(...)` qui sont l'API standard. Devrait exclure les fichiers de test ou les chaînes d'API fluent connues.
