# i18n-json-valid-message-syntax — supporter i18next nativement

**Type** : feature
**Priorité** : moyenne
**Source** : FP hunting session 1 (cal.com — 16096 FPs)

## Contexte

`i18n-json-valid-message-syntax` valide la syntaxe ICU MessageFormat (`{count}`, `{name, plural, ...}`). Mais beaucoup de projets utilisent i18next qui a une syntaxe différente :

- Interpolation : `{{count}}` au lieu de `{count}`
- Cross-reference : `$t(other_key)` 
- Pluralisation : suffixes de clé `_one`, `_other` au lieu de `{count, plural, one {...} other {...}}`
- Nesting : `$t(key, {"context": "value"})`

Actuellement on skip les strings contenant `{{` ou `$t(` — ça supprime les FPs mais on ne valide plus du tout la syntaxe i18next.

## Exemple

```json
// cal.com/apps/web/public/static/locales/fr/common.json
{
  "event_type_count": "{{count}} type d'événement",
  "event_type_count_other": "{{count}} types d'événement",
  "greeting": "$t(common:welcome), {{name}}!"
}
```

Avec ICU ces strings sont invalides (doubles accolades). Avec i18next c'est correct.

## Design proposé

1. Détecter le framework i18n du projet : `i18next` / `react-i18next` / `next-i18next` dans package.json → mode i18next
2. En mode i18next, valider :
   - `{{…}}` doit contenir un identifiant valide
   - `$t(…)` doit référencer une clé existante (optionnel, cross-file)
   - Les suffixes de pluralisation (`_zero`, `_one`, `_two`, `_few`, `_many`, `_other`) doivent être cohérents
3. En mode ICU (défaut), comportement actuel

## Projets impactés

cal.com, et probablement la majorité des projets Next.js/React qui utilisent i18next plutôt que FormatJS/react-intl.
