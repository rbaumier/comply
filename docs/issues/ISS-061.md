# TUI : panic possible sur highlight_cache index out-of-bounds

**Type** : bug (crash)
**Priorité** : haute
**Source** : recettage post-merge worktree-fp-hunting

## Problème

Dans `ui.rs:188-189`, le code indexe le cache de highlights avec `hl[i]` sans vérifier les bornes :

```rust
if let Some(hl) = app.highlight_cache.get(&index) {
    for (color, text) in &hl[i] {  // ← panic si i >= hl.len()
```

`i` provient de `context_lines.iter().enumerate()` (ligne 173). Le cache est construit dans `ensure_highlights()` (app.rs:233) qui appelle `source_lines()` indépendamment. Si le nombre de lignes retournées diffère entre l'appel dans `ensure_highlights` et celui dans `draw_preview` (fichier modifié entre-temps, LRU eviction, etc.), `hl[i]` panic.

## Scénario de reproduction

1. Ouvrir la TUI sur un projet
2. Modifier un fichier source pendant que la TUI est ouverte
3. Naviguer vers un diagnostic de ce fichier
4. Si le nombre de lignes du fichier a changé → crash

## Fix proposé

Remplacer l'indexation directe par un bounds check :

```rust
if let Some(hl) = app.highlight_cache.get(&index) {
    if let Some(spans) = hl.get(i) {
        for (color, text) in spans {
```

## Statut

✅ résolu — hl.get(i) remplace hl[i]

- [ ] La TUI ne crash plus si un fichier source change pendant la navigation
- [ ] Le highlighting fonctionne normalement
