# inverted-assertion-arguments FP sur Rust assert_eq!

**Règle** : `inverted-assertion-arguments`
**Impact** : 248 faux positifs sur tokio
**Priorité** : moyenne

## Problème

La règle suppose un ordre expected/actual dans les assertions (convention JUnit : `assertEquals(expected, actual)`). Mais `assert_eq!` en Rust n'a **aucun ordre conventionnel** — les deux positions sont symétriques. La macro affiche simplement "left" et "right" dans le message d'erreur.

```rust
// Les deux formes sont équivalentes et idiomatiques en Rust :
assert_eq!(0, map.len());
assert_eq!(map.len(), 0);
```

La règle ne devrait pas s'appliquer aux macros Rust `assert_eq!` / `assert_ne!`.

## Fix proposé

Désactiver `inverted-assertion-arguments` pour les fichiers `.rs`, ou au minimum pour les macros `assert_eq!` et `assert_ne!` qui sont symétriques par design.

## Fichiers concernés

- `src/rules/inverted_assertion_arguments/` — exclure les fichiers Rust ou les macros symétriques
