# rust-no-mutex-in-single-threaded : heuristique Arc trop simpliste

**Règle** : `rust-no-mutex-in-single-threaded`
**Impact** : 83 faux positifs sur tokio
**Priorité** : moyenne

## Problème

La règle détecte `Mutex<T>` sans `Arc<Mutex<T>>` et suggère `RefCell<T>`. L'heuristique suppose que si le Mutex n'est pas wrappé dans Arc, il n'est pas partagé entre threads. Mais c'est faux — un Mutex peut être partagé via :

- Un champ de struct qui est elle-même derrière un `Arc` (pattern très courant)
- Un `&self` dans un trait implémenté pour un type thread-safe
- Un `static` / `lazy_static` / `OnceLock`

Exemple tokio :
```rust
// tokio-util/src/sync/cancellation_token/tree_node.rs:47
inner: Mutex<Inner>,
// → la struct TreeNode est partagée via Arc ailleurs, pas ici
```

## Fix proposé

1. Ne pas flagguer `Mutex<T>` dans un champ de struct — le partage est déterminé par la struct parente
2. Ne pas flagguer `Mutex<T>` dans des `static` / `OnceLock` / `LazyLock`
3. Ne flagguer que les `Mutex<T>` locaux (variables dans une fonction) qui ne sont jamais passés à un `Arc::new()` ou `spawn`

## Fichiers concernés

- `src/rules/rust_no_mutex_in_single_threaded/` — affiner l'heuristique
