# Nouvelle règle : rust-prefer-arc-clone

**Règle proposée** : `rust-prefer-arc-clone`
**Type** : nouvelle règle
**Priorité** : basse

## Motivation

En Rust idiomatique, `Arc::clone(&x)` est préféré à `x.clone()` quand `x` est un `Arc<T>`. La raison : `.clone()` sur un Arc ne fait qu'incrémenter un compteur atomique (cheap), mais visuellement c'est indistinguable d'un `.clone()` sur un type qui fait une deep copy (expensive). `Arc::clone(&x)` rend l'intention explicite.

C'est l'équivalent de `clippy::clone_on_ref_ptr`, mais comply pourrait l'appliquer avec une meilleure heuristique (détection du type via tree-sitter + context).

Exemple :
```rust
// À éviter :
let handle = arc_ref.clone();

// Préféré :
let handle = Arc::clone(&arc_ref);
```

## Spécification

- Détecter `.clone()` sur des variables dont le type est `Arc<T>` (par inférence de la déclaration ou du pattern `let x: Arc<_>` / `let x = Arc::new(...)`)
- Ne pas flagguer `.clone()` sur des types qui implémentent Clone normalement
- Severity : `warning`
