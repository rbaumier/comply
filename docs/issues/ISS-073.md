# testing-no-undefined-mock-var FP sur les spy-only mocks

**Règle** : `testing-no-undefined-mock-var`
**Impact** : 66 faux positifs sur zustand
**Priorité** : moyenne

## Problème

La règle flagge `vi.fn()` sans `.mockReturnValue()` / `.mockImplementation()`, assumant que retourner `undefined` est involontaire. Mais beaucoup de mocks sont des **spies** — ils servent uniquement à traquer les appels via `.toHaveBeenCalled()`, pas à retourner des valeurs.

```typescript
const getItemSpy = vi.fn()
const setItemSpy = vi.fn()
// → utilisés uniquement avec expect(spy).toHaveBeenCalledWith(...)
```

Le retour `undefined` est parfaitement correct dans ce cas.

## Fix proposé

Ne pas flagguer un `vi.fn()` / `jest.fn()` non-configuré si, dans le même scope de test :
1. Il est uniquement utilisé avec des matchers de tracking d'appels (`.toHaveBeenCalled`, `.toHaveBeenCalledWith`, `.toHaveBeenCalledTimes`, `.toHaveBeenLastCalledWith`)
2. Sa valeur de retour n'est jamais lue (pas de `const result = spy()`, pas passé comme callback dont le retour est consommé)

Alternative plus simple : ne flagguer que si le mock est utilisé comme callback dont le retour est consommé (passé à `.map()`, `.filter()`, assigné à une variable).
