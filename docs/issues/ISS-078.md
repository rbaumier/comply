# dead-export / unused-file FP sur les exports publics de bibliothèque

**Règles** : `dead-export`, `unused-file`
**Impact** : 974 + 552 = 1526 faux positifs sur redux-toolkit
**Priorité** : haute

## Problème

Les deux règles analysent le graphe d'imports **interne** au projet. Pour une bibliothèque (npm package), les exports sont consommés par des projets **externes** — pas par le code du package lui-même.

Exemples flaggés à tort sur redux-toolkit :
```
packages/toolkit/src/createSlice.ts     → "unused-file" (fichier core de RTK!)
packages/toolkit/src/configureStore.ts  → "unused-file"
packages/toolkit/src/matchers.ts        → "unused-file"
```

Tout le dossier `packages/toolkit/src/` est flaggé car ces fichiers sont exportés via `index.ts` pour les consommateurs, mais ne s'importent pas les uns les autres.

Aussi impacté : les `__testfixtures__/` (fichiers de fixture pour codemods) qui sont des fichiers de test standalone par design.

## Fix proposé

1. Détecter les projets **bibliothèque** : présence de `"main"`, `"exports"`, ou `"module"` dans `package.json`
2. Pour les bibliothèques, traiter les fichiers référencés par `package.json#exports` ou `index.ts` comme des entry points publics
3. Ne pas flagguer les exports de ces fichiers comme "dead" — ils sont l'API publique
4. Exclure les `__testfixtures__/`, `__fixtures__/`, `fixtures/` qui sont des fichiers de test standalone

## Fichiers concernés

- `src/rules/dead_export/` — ajouter détection bibliothèque
- `src/rules/unused_file/` — ajouter entry points depuis package.json exports
- `src/engine/` — potentiellement le graph builder pour les entry points
