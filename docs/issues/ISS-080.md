# filename-naming-convention impose kebab-case sur les composants React (PascalCase)

**Règle** : `filename-naming-convention`
**Impact** : 370 faux positifs sur redux-toolkit
**Priorité** : moyenne
**Statut** : résolu — l'implémentation accepte maintenant kebab-case, PascalCase et camelCase pour les fichiers TS/JS.

## Problème

La règle impose kebab-case pour tous les fichiers. Mais les conventions TS/React sont :

- **PascalCase** pour les fichiers de composants React (`App.tsx`, `PostsManager.tsx`, `BraveWarning.js`)
- **camelCase** pour les modules TS utilitaires (`createSlice.ts`, `dynamicMiddleware.ts`, `listenerMiddleware.ts`)
- **kebab-case** n'est la convention que pour les fichiers non-code (CSS, assets) ou certains frameworks (Angular)

La communauté React a standardisé PascalCase = composant depuis sa création. Forcer kebab-case va à l'encontre de toute la documentation React, Next.js, et de la majorité des projets OSS.

## Fix proposé

Rendre la convention configurable par type de fichier :
1. `.tsx`/`.jsx` → accepter PascalCase (composants React) ET kebab-case
2. `.ts`/`.js` → accepter camelCase ET kebab-case
3. `index.ts`/`index.js` → toujours OK (convention universelle)
4. Ou détecter la convention dominante du projet et s'y conformer

Alternative : si React est détecté (framework detection), passer automatiquement en mode PascalCase-allowed pour `.tsx`/`.jsx`.

## Fichiers concernés

- `src/rules/filename_naming_convention/` — rendre configurable ou détecter le framework

## Review 2026-05-01

- `src/rules/filename_naming_convention/text.rs` accepte bien PascalCase pour les fichiers TS/JS/TSX/JSX.
- Le même fichier garde explicitement `flags_camel_case_ts`, donc `createSlice.ts`, `dynamicMiddleware.ts` et `listenerMiddleware.ts` restent des faux positifs.
- Le fix demandé aux lignes 20-22 n'est donc que partiellement implémenté.

## Résolution 2026-05-01

- Ajout de l'acceptation camelCase pour les fichiers `.ts`, `.tsx`, `.js` et `.jsx`.
- Les tests de régression encodent maintenant `userProfile.ts`, `dynamicMiddleware.js` et `externalLinks.ts` comme valides, tout en gardant les rejets snake-case/dash invalides.
