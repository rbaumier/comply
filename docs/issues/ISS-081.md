# exports-last FP sur les inline `export const/function`

**Règle** : `exports-last`
**Impact** : 414 faux positifs sur redux-toolkit
**Priorité** : moyenne

## Problème

La règle veut que tous les exports soient en fin de fichier. Mais le pattern dominant en TypeScript moderne est l'**export inline** :

```typescript
export const container = document.getElementById('root')
export function createSlice(config) { ... }
export type SliceState = { ... }
```

Ce pattern est recommandé par la documentation TypeScript, utilisé dans tous les projets OSS majeurs, et est le style par défaut de la plupart des formateurs/linters.

La convention "exports-last" vient de CommonJS (`module.exports = { ... }` en fin de fichier) et des re-export barrels. Elle n'est pas applicable aux inline exports ES modules.

## Fix proposé

Distinguer deux types d'exports :
1. **Inline exports** (`export const`, `export function`, `export class`, `export type`, `export interface`) → ne pas flagguer
2. **Re-exports** (`export { x, y }`, `export { x } from './mod'`) → appliquer la règle (ceux-ci doivent être groupés)

Ou simplement : ne pas appliquer la règle quand le fichier n'a que des inline exports.
