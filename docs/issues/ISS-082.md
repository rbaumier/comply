# file-extension-in-import FP quand un bundler est configuré

**Règle** : `file-extension-in-import`
**Impact** : 909 faux positifs sur redux-toolkit
**Priorité** : moyenne
**Statut** : résolu — la règle lit maintenant `compilerOptions.moduleResolution` et détecte aussi les configs bundler courantes.

## Problème

La règle exige des extensions explicites sur les imports relatifs pour la compatibilité ESM Node.js. Mais la grande majorité des projets TS utilisent un **bundler** (webpack, vite, esbuild, turbopack) qui résout les extensions automatiquement.

```typescript
import { App } from './App'        // → flaggé, mais webpack/vite résout vers App.tsx
import { store } from './app/store' // → flaggé, mais le bundler gère
```

Exiger des extensions dans un projet bundlé :
- Ajoute du bruit sans valeur (le bundler gère)
- Peut casser la résolution (`.ts` n'est pas une extension valide à l'import dans certains configs)
- Ne correspond pas aux conventions de l'écosystème React/Vue/Svelte

## Fix proposé

Détecter la présence d'un bundler et désactiver la règle :
1. `webpack.config.*` / `next.config.*` → bundler détecté
2. `vite.config.*` → bundler détecté
3. `tsconfig.json` avec `"moduleResolution": "bundler"` → explicitement configuré
4. Présence de `@vitejs/*`, `webpack`, `next`, `turbopack` dans les deps

Si un bundler est détecté, ne pas appliquer `file-extension-in-import`.

## Fichiers concernés

- `src/rules/file_extension_in_import/` — ajouter détection bundler
- `src/frameworks/mod.rs` — potentiellement enrichir la détection de framework

## Review 2026-05-01

- `src/rules/file_extension_in_import/typescript.rs` cherche bien les dépendances bundler (`vite`, `webpack`, `esbuild`, etc.).
- En revanche, `tsconfig_uses_bundler_resolution` compare `ts.module` à `"bundler"`.
- `src/project/mod.rs` parse `compilerOptions.module`, pas `compilerOptions.moduleResolution`. Une config valide avec `"moduleResolution": "bundler"` ne désactive donc pas la règle.
- Les signaux `webpack.config.*`, `vite.config.*` et `next.config.*` ne sont pas directement détectés si la dépendance correspondante n'est pas visible dans le `package.json` le plus proche.

## Résolution 2026-05-01

- `Tsconfig` parse et merge maintenant `compilerOptions.moduleResolution`.
- `file-extension-in-import` utilise ce champ pour `moduleResolution: "bundler"` au lieu de `module`.
- La règle détecte aussi `vite.config.*`, `webpack.config.*`, `next.config.*`, `turbopack.config.*`, `next` et les dépendances `@vitejs/*`.
