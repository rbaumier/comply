# no-type-encoded-names FP sur préfixes de domaine (str_, arr_)

**Règle** : `no-type-encoded-names`
**Impact** : 235 faux positifs sur polars
**Priorité** : moyenne

## Problème

La règle détecte la notation hongroise (préfixes de type : `str_`, `int_`, `arr_`). Mais dans polars, ces préfixes encodent le **domaine**, pas le type :

```rust
let str_s = s.cast(&DataType::String)?;
// → str_ signifie "colonne castée en String", pas "variable de type &str"

let arr_lhs = ...;
// → arr_ signifie "version Array du LHS", pas "variable de type Array"
```

C'est une convention de nommage du domaine data (string column, array column) pas du typage. Sans le préfixe, `s` et `str_s` seraient indistinguables alors qu'ils désignent des colonnes de types différents.

## Fix proposé

1. Ne pas flagguer les préfixes qui correspondent à des **noms de domaine** courants dans l'écosystème data : `str_`, `int_`, `float_`, `bool_`, `arr_`, `list_`, `cat_`, `date_`, `time_`
2. Ou ne flagguer que quand le préfixe correspond exactement au type Rust de la variable (vrai Hungarian notation)
3. Alternative : ne flagguer que les préfixes qui encodent des types Rust primitifs (`i32_`, `u64_`, `f64_`, `vec_`, `string_`)
