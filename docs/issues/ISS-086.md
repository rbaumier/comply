# rust-no-as-numeric-cast trop strict pour le code data-processing

**Règle** : `rust-no-as-numeric-cast`
**Impact** : 3160 faux positifs sur polars
**Priorité** : haute
**Statut** : résolu — la règle distingue maintenant les widening casts sûrs des casts dangereux quand le type source est localement connu.

## Problème

La règle interdit `as usize`, `as f64`, etc. et suggère `From`/`TryFrom`. Mais dans une bibliothèque de data processing comme polars, les casts numériques sont **l'opération fondamentale** :

```rust
// Chaque transformation de données implique des casts
let idx = row_idx as usize;
let val = raw_bytes as f64;
```

Utiliser `try_from` partout :
- Rend le code illisible (noise de `.try_from().unwrap()` partout)
- Ralentit les hot paths (check de range inutile quand le domaine est connu)
- N'apporte pas de sécurité quand les bornes sont garanties par invariant

3160 violations sur un seul projet montre que la règle n'est pas calibrée pour ce domaine.

## Fix proposé

Options par ordre de préférence :
1. **Seuil de tolérance** : si un crate a plus de N `as` casts (ex: 50), réduire la sévérité à `info` — c'est clairement un projet numérique
2. **Heuristique domaine** : détecter les crates data/math (dépendances `ndarray`, `arrow`, `parquet`, `num-traits`) et relâcher
3. **Scope** : ne flagguer que les casts "dangereux" (`as` entre signed/unsigned, narrowing) et ignorer les widening casts sûrs (`u32 as u64`)
4. **Configurable** : permettre dans `comply.toml` de désactiver la règle par crate

L'option 3 est la plus universelle : les widening casts (`u32 as usize`, `i32 as i64`) sont toujours sûrs et n'ont pas besoin de `From`.

## Fichiers concernés

- `src/rules/rust_no_as_numeric_cast/` — distinguer widening vs narrowing casts

## Review 2026-05-01

- `src/rules/rust_no_as_numeric_cast/rust.rs` retourne tôt pour `as usize`, `as isize`, les casts de littéraux et le contexte de test.
- Le code continue de flagguer `u8 as u64`, et le test `flags_widening_u8_to_u64` encode ce comportement.
- Cela contredit l'option 3 retenue dans cette fiche : ignorer les widening casts sûrs et concentrer la règle sur les casts dangereux.
- Aucune heuristique data/math ni seuil de tolérance par crate n'est présent.

## Résolution 2026-05-01

- Ajout d'une inférence locale pour les paramètres et `let` typés.
- Les widening casts entiers à signedness identique (`u8 as u64`, `i32 as i64`) ne sont plus flaggués.
- Les casts narrowing, signed/unsigned, float et sources inconnues restent signalés de manière conservatrice.
