# rust-pub-enum-without-non-exhaustive FP sur enums workspace-internal

**Règle** : `rust-pub-enum-without-non-exhaustive`
**Impact** : 368 faux positifs sur polars
**Priorité** : basse
**Statut** : résolu — la règle distingue maintenant les crates internes de workspace des crates explicitement publiées.

## Problème

La règle exige `#[non_exhaustive]` sur tout `pub enum`. Mais dans un workspace Cargo, `pub` signifie souvent "visible pour les crates sœurs", pas "API publique pour les consommateurs externes".

```rust
// polars-expr/src/expressions/mod.rs
pub enum AggState { ... }
pub enum UpdateGroups { ... }
// → utilisés uniquement par polars-mem-engine et polars-plan, pas par les utilisateurs
```

`#[non_exhaustive]` sur ces enums serait nuisible : il forcerait un `_ => unreachable!()` dans chaque match, même dans le code interne qui contrôle tous les variants.

## Fix proposé

1. Ne pas flagguer les `pub enum` dans les crates qui ne sont pas le crate racine du package (crates avec `publish = false` ou sans `[package]` section publique)
2. Ne pas flagguer les `pub(crate)` enums (déjà le cas ?)
3. Détecter les workspace members et ne flagguer que les enums dans les crates qui ont `publish = true` ou qui sont dans `package.json#exports`

## Fichiers concernés

- `src/rules/rust_pub_enum_without_non_exhaustive/` — ajouter détection workspace/publish

## Review 2026-05-01

- `src/rules/rust_pub_enum_without_non_exhaustive/rust.rs` ignore les crates dont le `Cargo.toml` contient `publish = false` ou `publish = []`.
- La fiche demande aussi de détecter les workspace members et de distinguer les crates réellement exposées des crates internes.
- Il n'y a pas de lecture structurée du manifeste Cargo, ni de détection du workspace root/member, ni de test couvrant un crate interne de workspace.
- Les `pub(crate)` et `pub(super)` sont déjà évités par `is_pub`, mais le coeur du faux positif Polars reste partiellement traité seulement.

## Résolution 2026-05-01

- Lecture structurée de `Cargo.toml` via TOML au lieu d'une recherche texte.
- Les manifests `publish = false`, `publish = []`, les manifests sans `[package]` et les membres de workspace sans publication explicite sont traités comme internes.
- Ajout de régressions pour `publish = false`, workspace member interne, `publish = true` et `publish = ["registry"]`.
