# Crash UTF-8 byte-indexing systémique dans plusieurs règles

**Impact** : crash (panic) — bloque l'analyse sur tout projet avec des caractères multi-byte
**Priorité** : critique
**Lié à** : ISS-072 (même cause racine)

## Problème

Le bug UTF-8 d'ISS-072 n'est pas limité à `api-response-envelope-consistency`. Au moins deux règles sont touchées :

### 1. `api-response-envelope-consistency` (trpc)
```
byte index 62 is not a char boundary; it is inside '💡' (bytes 61..65)
at typescript.rs:114:24
```

### 2. `zod-no-safeparse-without-check` (zod)
```
byte index 1005 is not a char boundary; it is inside 'ვ' (bytes 1003..1006)
at typescript.rs:79:33
```

### 3. `dead_export` (date-fns)
```
byte index 23 is not a char boundary; it is inside 'ل' (bytes 22..24)
at text.rs:48:23
```

Les emoji (4 bytes), les caractères géorgiens (3 bytes), arabes (2 bytes), cyrilliques, CJK, etc. causent tous le même panic. La cause est un pattern commun : `&text[..n]` ou `text.as_bytes()[n]` sans vérifier les char boundaries.

**3 règles confirmées sur 3 projets différents** — le bug est systémique et touche potentiellement toute règle qui slice du texte source.

## Fix proposé

1. **Audit systématique** : grep tout le code pour les patterns `&text[`, `text[..`, `.as_bytes()[` et vérifier qu'ils respectent les boundaries UTF-8
2. **Fix global** : introduire un helper `safe_slice(text, byte_start, byte_end)` qui clamp aux char boundaries
3. **Tests de régression** : ajouter des fichiers de test avec des emoji, CJK, cyrillique, géorgien, arabe dans les commentaires et les strings

## Fichiers concernés

- `src/rules/api_response_envelope_consistency/typescript.rs:114`
- `src/rules/zod_no_safeparse_without_check/typescript.rs:79`
- `src/rules/dead_export/text.rs:48`
- Potentiellement toutes les règles qui manipulent du texte source
