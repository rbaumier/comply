# no-unsafe-* rules ne devraient pas fire sur les fichiers .js purs

**Règles** : `no-unsafe-call`, `no-unsafe-member-access`, `no-unsafe-assignment`, `no-unsafe-return`, `no-unsafe-argument`
**Impact** : 36745 faux positifs sur fastify (19036 + 17709 + 3383 + 491 + 581)
**Priorité** : critique

## Problème

Les règles `no-unsafe-*` vérifient l'utilisation de valeurs typées `any` en TypeScript. Mais sur les fichiers `.js` **sans annotations de type**, toute expression est implicitement `any`. Le résultat : chaque appel de fonction, chaque accès de propriété, chaque assignation est flaggée.

```javascript
// fastify.js — pur JS
const http = require('http')  // → no-unsafe-call, no-unsafe-assignment
http.createServer()           // → no-unsafe-call, no-unsafe-member-access
```

36745 violations sur un seul projet JS montre que ces règles ne sont pas conçues pour le JavaScript pur.

## Fix proposé

Ne pas appliquer les règles `no-unsafe-*` sur les fichiers `.js` et `.mjs` :
- Ces règles n'ont de sens qu'avec des annotations de type ou des inférences TypeScript
- Sur du JS pur, elles flagguent littéralement tout le code
- Si un projet utilise JSDoc pour le typage, la détection pourrait être affinée (mais c'est rare)

## Fichiers concernés

- `src/rules/no_unsafe_call/`, `no_unsafe_member_access/`, `no_unsafe_assignment/`, `no_unsafe_return/`, `no_unsafe_argument/` — ajouter filtre d'extension `.ts`/`.tsx` only
