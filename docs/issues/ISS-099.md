# timeout-on-io FP sur les internals de framework et test clients

**Règle** : `timeout-on-io`
**Impact** : 268 faux positifs sur axum
**Priorité** : moyenne

## Problème

La règle demande un timeout sur chaque appel I/O. Mais elle fire sur :

### 1. Framework internals
```rust
// axum/src/handler/mod.rs — le framework gère ses propres timeouts via tower middleware
let res = client.get("/").await;  // ← framework internal test
```
Axum est le framework — ses internals ne doivent pas wrapper chaque `.await` d'un timeout car c'est à l'utilisateur de configurer les timeouts via `tower::timeout::TimeoutLayer`.

### 2. Test clients
```rust
// Tests qui appellent un serveur in-process
let client = TestClient::new(svc);
let res = client.get("/").await;  // ← serveur local, pas de "slow peer"
```

## Fix proposé

1. Ne pas flagguer les `.await` dans les fichiers de test
2. Ne pas flagguer les appels I/O dans les crates qui SONT des frameworks (détection : le crate exporte des `Router`, `Service`, `Handler` types)
3. Ou se concentrer sur les appels réseau explicites (`reqwest::get`, `hyper::Client`, `TcpStream::connect`) plutôt que tout `.await`

## Fichiers concernés

- `src/rules/timeout_on_io/` — affiner la détection d'appels réseau
