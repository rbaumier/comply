# Plusieurs règles devraient être relâchées dans examples/ et benches/

**Règles** : `rust-serde-deny-unknown-fields`, `rust-anyhow-context-on-question-mark`, `structured-api-error`, et autres
**Impact** : ~175 faux positifs sur axum (79 + 57 + 39)
**Priorité** : moyenne

## Problème

Les dossiers `examples/` et `benches/` ont un objectif différent du code de production :

### examples/
Le but est la **lisibilité** et la **simplicité**. Ajouter `.context("...")` à chaque `?`, `#[serde(deny_unknown_fields)]` à chaque struct, ou des error types structurés rend le code d'exemple verbeux et difficile à apprendre :

```rust
// examples/jwt/src/main.rs
let token = encode(&Header::default(), &claims, &key)?;
// → flaggé: "Add .context() before ?" — mais c'est un exemple de 30 lignes
```

### benches/
Le but est la **mesure de performance**. Les benchmarks utilisent des structs minimales et `panic!` car les erreurs ne sont pas le sujet :

```rust
// benches/benches.rs
#[derive(Deserialize)]
struct Payload { n: u32, s: String, b: bool }
// → flaggé: "missing deny_unknown_fields" — c'est un bench, pas une API
```

### rust-serde-deny-unknown-fields sur les types de bibliothèque
Indépendamment des exemples, `deny_unknown_fields` sur les types publics d'une bibliothèque est souvent **wrong** — cela empêche la forward compatibility (un champ ajouté par le serveur casse le client).

## Fix proposé

1. Réduire la sévérité (ou ignorer) pour les fichiers dans `examples/`, `benches/`, `fixtures/`
2. Pour `rust-serde-deny-unknown-fields` : ne flagguer que les types d'input HTTP (API boundary), pas les types internes ou de bibliothèque
3. Ajouter un concept global `relaxed_directories` dans la config pour définir les dossiers "non-production"

## Fichiers concernés

- `src/engine/mod.rs` — concept de "relaxed directories"
- `src/rules/rust_serde_deny_unknown_fields/` — scoper aux API boundaries
- `src/rules/rust_anyhow_context_on_question_mark/` — skip examples/
