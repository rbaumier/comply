# ts-no-extraneous-class ne détecte pas les décorateurs sur `export class`

**Règle** : `ts-no-extraneous-class`, `no-extraneous-class`
**Impact** : ~1024 faux positifs sur nest
**Priorité** : haute

## Problème

La règle skip les classes décorées (ligne 16-21 de `typescript.rs`), mais seulement quand le décorateur est un enfant direct du nœud `class_declaration`. Or pour `export class`, tree-sitter structure l'AST ainsi :

```
export_statement
  decorator        ← le décorateur est ICI
  class_declaration
    type_identifier
    class_body
```

Le décorateur est un enfant de `export_statement`, pas de `class_declaration`. Le skip ne le voit donc pas.

### Exemples flaggés à tort

```typescript
// tools/benchmarks/src/frameworks/nest/app.module.ts
@Module({
  imports: [],
  controllers: [AppController],
})
export class AppModule {}  // → "Unexpected empty class" — c'est un module NestJS

// integration/inspector/src/dogs/dto/create-dog.dto.ts
@ApiProperty()
export class CreateDogDto {}  // → "Unexpected empty class" — c'est un DTO

// integration/inspector/src/dogs/dogs.module.ts
@Module({ controllers: [DogsController] })
export class DogsModule {}  // → idem
```

Ce pattern est le standard absolu pour NestJS, Angular, et tout framework à décorateurs TypeScript.

## Fix proposé

En plus de checker les children du nœud `class_declaration`, checker aussi le parent (`export_statement`) pour des enfants `decorator` :

```rust
// Aussi vérifier le parent (export_statement) pour les décorateurs
if let Some(parent) = node.parent() {
    let mut pc = parent.walk();
    for child in parent.children(&mut pc) {
        if child.kind() == "decorator" {
            return;
        }
    }
}
```

## Fichiers concernés

- `src/rules/ts_no_extraneous_class/typescript.rs` — ajouter check parent pour decorator
- La règle `no-extraneous-class` (backend séparé) a probablement le même bug
