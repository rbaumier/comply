# parameter-properties FP sur l'injection de dépendances

**Règles** : `parameter-properties`, `ts-parameter-properties`
**Impact** : ~774 faux positifs sur nest
**Priorité** : haute

## Problème

Les deux règles flagguent les "parameter properties" TypeScript (`private readonly x: T` dans un constructeur) en recommandant de déclarer la propriété explicitement. Mais c'est le pattern canonical pour l'injection de dépendances dans NestJS, Angular, et TypeORM :

```typescript
// packages/microservices/src/nats/nats.controller.ts
@Controller()
export class NatsController {
  constructor(private readonly natsService: NatsService) {}
  //          ↑ flaggé "should be declared as a class property"
}
```

Ce pattern est :
- Recommandé par la documentation officielle NestJS
- Requis par le système de DI (le décorateur `@Inject()` implicite repose sur les metadata du paramètre)
- Le seul moyen idiomatique de faire de la DI en TypeScript

Déclarer la propriété séparément serait du boilerplate inutile et briserait le pattern DI.

## Fix proposé

Ne pas flagguer les parameter properties dans les classes décorées :
1. Si la classe a un décorateur (`@Injectable`, `@Controller`, `@Module`, etc.), skip les parameter properties du constructeur
2. Alternative : ne pas flagguer quand le parameter property a `private readonly` (pattern DI universel)

## Fichiers concernés

- `src/rules/parameter_properties/` — skip dans classes décorées
- `src/rules/ts_parameter_properties/` — idem
