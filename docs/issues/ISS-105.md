# no-async-without-await FP sur handlers de framework contractuels

**Règle** : `no-async-without-await`
**Impact** : ~460 faux positifs sur nest
**Priorité** : moyenne

## Problème

La règle flagge les fonctions `async` qui n'utilisent pas `await`. Mais dans NestJS (et d'autres frameworks), certains handlers DOIVENT être `async` car le framework attend un `Promise<>` :

### 1. gRPC handlers
```typescript
@GrpcMethod('Math')
async sum({ data }: { data: number[] }): Promise<any> {
  return of({ result: data.reduce((a, b) => a + b) });
}
// → le framework gRPC requiert Promise<any>, async est obligatoire
```

### 2. Lifecycle hooks
```typescript
async onModuleInit(): Promise<void> {
  this.connection = createConnection();
}
// → l'interface OnModuleInit attend Promise<void>
```

### 3. Test setup/teardown
```typescript
it('should work', async () => {
  return request(server).get('/').expect(200);
});
// → return promise, async pour le type-checker
```

## Fix proposé

1. Ne pas flagguer les méthodes `async` dont le type de retour explicite est `Promise<>` (le contrat est explicite)
2. Ne pas flagguer les méthodes dans des classes décorées (framework contract implicite)
3. Ne pas flagguer dans les fichiers de test (déjà proposé ailleurs)

## Fichiers concernés

- `src/rules/no_async_without_await/` — exceptions pour Promise return type et classes décorées
