# no-class-inheritance FP sur les points d'extension de framework

**Règle** : `no-class-inheritance`
**Impact** : ~336 faux positifs sur nest
**Priorité** : moyenne

## Problème

La règle recommande la composition plutôt que l'héritage. C'est un bon conseil en général, mais les frameworks DI fournissent des classes de base conçues pour être étendues :

```typescript
// packages/core/exceptions/base-exception-filter-context.ts
export class BaseExceptionFilterContext extends ContextCreator {
  // → NestJS fournit ContextCreator pour être étendu
}

// packages/microservices/server/server-grpc.ts
export class ServerGrpc extends Server {
  // → Server est la classe de base pour les transports
}

// integration/microservices/src/app.module.ts
class ErrorHandlingProxy extends ClientTCP {
  // → ClientTCP est conçu pour être sous-classé
}
```

NestJS utilise l'héritage comme mécanisme d'extension principal pour :
- Exception filters (`BaseExceptionFilter`)
- Guards (`AuthGuard`)
- Interceptors (`CacheInterceptor`)
- Transports (`Server`, `ClientProxy`)
- Adapters (`AbstractHttpAdapter`)

## Fix proposé

1. Ne pas flagguer l'héritage quand la superclasse est importée d'un package `@nestjs/*`, `@angular/*`, ou similaire
2. Alternative : ne pas flagguer quand la superclasse est une classe abstraite (détectable via `abstract` keyword)
3. Alternative minimale : ne pas flagguer dans les fichiers de framework (`*.module.ts`, `*.guard.ts`, `*.interceptor.ts`, `*.filter.ts`)

## Fichiers concernés

- `src/rules/no_class_inheritance/` — exceptions pour classes de base de framework
