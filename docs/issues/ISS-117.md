# no-side-effects-in-initialization FP sur frameworks réactifs (Solid, Vue)

**Règle** : `no-side-effects-in-initialization`
**Impact** : ~224 faux positifs sur solid
**Priorité** : haute

## Problème

La règle flagge les side effects au top-level des modules. Mais les frameworks réactifs comme SolidJS (et Vue Composition API) reposent ENTIÈREMENT sur les side effects à l'initialisation :

```tsx
// SolidJS — c'est COMMENT le framework fonctionne
function Counter() {
  const [count, setCount] = createSignal(0)  // ← side effect (réactif)
  createEffect(() => console.log(count()))    // ← side effect (tracking)
  const doubled = createMemo(() => count() * 2)  // ← side effect (computed)
  return <button onClick={() => setCount(c => c + 1)}>{doubled()}</button>
}
```

`createSignal`, `createEffect`, `createMemo`, `createResource`, `onMount`, `onCleanup` sont tous des side effects d'initialisation. C'est le paradigme fondamental de Solid — sans ces appels, le composant n'a aucune réactivité.

Ce pattern est aussi présent dans Vue 3 Composition API (`ref()`, `computed()`, `watch()`, `onMounted()`).

## Fix proposé

1. Détecter les imports de frameworks réactifs (`solid-js`, `vue`) et skip la règle sur les fichiers qui les importent
2. Ou whitelister les fonctions de création de signaux/effects connues (`createSignal`, `createEffect`, `ref`, `computed`, etc.)
3. Limiter la règle aux modules non-composants (fichiers sans JSX et sans imports de framework réactif)
