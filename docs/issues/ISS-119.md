# zod-validate-env-at-startup FP sur le fichier de validation env lui-même

**Règle** : `zod-validate-env-at-startup`
**Impact** : ~48 faux positifs sur create-t3-app
**Priorité** : haute

## Problème

La règle demande de valider `process.env` via Zod au démarrage. Mais elle flagge les fichiers `env.js`/`env.ts` qui SONT la validation Zod :

```javascript
// src/env.js — c'est LE fichier de validation env
import { createEnv } from "@t3-oss/env-nextjs"
import { z } from "zod"

export const env = createEnv({
  server: { DATABASE_URL: z.string() },
  client: { NEXT_PUBLIC_APP_URL: z.string() },
  runtimeEnv: {
    DATABASE_URL: process.env.DATABASE_URL,      // ← flaggé !
    NEXT_PUBLIC_APP_URL: process.env.NEXT_PUBLIC_APP_URL,  // ← flaggé !
  },
})
```

Le fichier fait exactement ce que la règle demande (validation Zod) mais les accès `process.env.*` dans le `runtimeEnv` sont quand même flaggés. 48/48 violations sont dans des fichiers `/env`.

## Fix proposé

1. Détecter les imports de `@t3-oss/env-*`, `createEnv`, ou les appels `z.parse(process.env)` dans le même fichier
2. Si le fichier contient une validation Zod de process.env, ne pas flagguer les accès process.env dans ce fichier
