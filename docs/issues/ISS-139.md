# id-length trop strict sur variables d'itération et destructuring

**Règle** : `id-length`
**Impact** : FP #1 cross-projet — 14348 fp-ts, 1102 tokio, 836 kanidm, 826 jiff, 806 rayon, 774 documenso, 628 serde, 605 ts-pattern, 475 firecracker, 375 rustls, 349 helix, 307 ripgrep, 155 hyper, 117 svix
**Priorité** : critique

## Problème

La règle interdit les identifiants de moins de 2 caractères. Mais plusieurs patterns Rust idiomatiques utilisent des noms courts :

```rust
// 1. Closures et itérateurs — le contexte donne le sens
items.iter().filter(|x| x.is_valid())
map.iter().for_each(|(k, v)| println!("{k}: {v}"))

// 2. Destructuring de tuples
let (x, y) = point.coordinates();

// 3. Variables mathématiques
let n = items.len();
let i = offset + 1;

// 4. Conventionnel Rust : `e` pour error dans map_err
result.map_err(|e| Error::Wrap(e))
```

ISS-116 avait corrigé les paramètres de closure mais le problème persiste pour les closures d'itérateur et le destructuring.

## Fix proposé

1. Exclure les variables dans les closures passées à `.map()`, `.filter()`, `.for_each()`, `.map_err()`, etc.
2. Exclure les variables de destructuring de tuples
3. Autoriser les noms mathématiques courants : `n`, `i`, `j`, `k`, `x`, `y`, `z`, `e`
4. Exclure les variables dans les closures de 1-3 lignes
