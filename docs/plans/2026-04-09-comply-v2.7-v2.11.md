# comply v2.7 → v2.11 — Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Combler le gap entre les règles syntaxiques actuellement implémentées (~87 règles AST file-scoped) et la couverture complète de la skill `coding-standards` (~130 règles, dont une majorité sémantiques). Progression batchée v2.7→v2.11, chaque version shippable seule.

**Stratégie globale :** "Comply couvre tout. Les règles impossibles à juger syntaxiquement passent par un backend LLM." Trois couches :
1. **AST file-scoped** (Rust + tree-sitter, in-process) — Cat A + Comments + Naming
2. **Délégation à des runners externes** (jscpd, knip/cargo-shear, madge/cargo-modules) pour le cross-file
3. **LLM backend opt-in** (`--with-llm`) via `claude-code-proxy` en sidecar pour les règles sémantiques

## Decisions Log (grilling session 2026-04-09)

| Décision | Choix | Rationale |
|---|---|---|
| Scope global | Couverture totale, LLM pour le sémantique | "On avance progressivement jusqu'à ce but" |
| FP par règle | Configurable via champ `max_fp_rate` dans `Rule` | 0% strict pour règles AST safe, budget > 0 pour heuristiques |
| Comportement CI | `max_fp_rate == 0` → error (bloque), `> 0` → warning | Évite les faux blocs sur les règles à seuil |
| Mesure FP | Corpus de référence par règle dans `tests/fp_corpus/<rule_id>.{ts,rs}` | Tests automatisés, mesure objective |
| Cross-file analysis | Délégation à outils externes (jscpd/knip/madge/cargo-shear/cargo-modules) | Pattern déjà éprouvé avec clippy/oxlint runners |
| LLM backend | `claude-code-proxy` en sidecar HTTP, `reqwest` + tokio worker pool | Respecte ToS Anthropic Feb 2026, parallélisable |
| Modèle LLM par défaut | Sonnet 4.6, override via `--model haiku` | Précision > coût pour les règles sémantiques |
| Cache LLM | SQLite à 2 niveaux : `(file_sha256, rule_id)` + `(block_sha256, rule_id)` | Granularité fine |
| Cache invalidation | Aucun TTL, hash-based uniquement | Déterministe |
| Définition d'un "bloc" | AST node top-level (fn, struct, type, const exporté) | Aligné sur le parser tree-sitter |
| Path du cache SQLite | `.comply/cache.db` per-projet | Évite collisions cross-projet |
| Versionnement règles LLM | `const RULE_VERSION: u32` par règle, inclus dans hash de cache | Bump = invalidation auto |
| Flag CLI | `comply --with-llm` | Explicite, opt-in |
| Fallback CI sans LLM | `panic!` si `--with-llm` et proxy indisponible | Pas de skip silencieux |
| Shotgun Surgery / Divergent Change | **Backlog v3**, abandon v2 | Nécessite git history + outil dédié |
| Ordre de release | v2.7 → v2.11, batchs versionnés indépendants | Chaque version shippable seule |

## Risques connus

| Risque | Impact | Mitigation |
|---|---|---|
| FP=0% strict bloque des règles utiles | Moyen | `max_fp_rate` configurable, warning-mode |
| `claude-code-proxy` cassé par update Anthropic | Élevé sur v2.11 | Pin version, fallback `--no-llm`, tests nightly |
| Runners externes divergent en formats JSON | Moyen sur v2.10 | Pin versions, integration tests |
| Cache LLM grossit indéfiniment | Faible | `comply cache prune --older-than 30d` |
| Worker pool LLM saturé | Moyen sur v2.11 | Backpressure channel borné |
| Auto-dogfood échoue après ajout v2.7 | Faible | TDD strict, corpus FP avant impl |
