# comply v2.11 — LLM Backend Design

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

## Overview

Add an opt-in LLM backend (`--with-llm`) that spawns the `claude` CLI as a subprocess to evaluate semantic rules that tree-sitter can't detect mechanically. Uses the local Claude subscription (no API key, no extra cost).

## Architecture

```
comply --with-llm src/
    │
    ├── Phase 1: AST rules (in-process, unchanged)
    ├── Phase 2: External runners (clippy, oxlint, jscpd, etc.)
    └── Phase 3: LLM rules (new)
            │
            ├── For each file × each LLM rule:
            │     1. Extract blocks (fn/struct/type/comment)
            │     2. Hash block → cache key (sha256 of rule_id + rule_version + content)
            │     3. Cache hit? → return cached diagnostic
            │     4. Cache miss → spawn `claude -p --output-format json --json-schema <schema>`
            │     5. Parse JSON response → Vec<Diagnostic>
            │     6. Store in SQLite cache
            │
            └── Concurrency: tokio::sync::Semaphore(8) limits parallel subprocesses
```

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| LLM invocation | `claude` CLI subprocess (`-p --output-format json`) | Uses local subscription (free), structured output via `--json-schema`, already installed |
| No Rig/SDK | Direct subprocess | Zero deps, subscription locale, CLI handles auth+rate-limiting |
| Cache | SQLite `.comply/cache.db` per-project | Hash-based, no TTL, deterministic invalidation |
| Cache key | `sha256(rule_id + rule_version + block_sha256)` | Rule version bump = auto-invalidation |
| Block definition | AST top-level node (fn, struct, type, const) | Aligned with tree-sitter, granular cache |
| Concurrency | `tokio::sync::Semaphore(8)` | Prevents spawning 100 claude processes |
| Model | Sonnet by default, `--model haiku` override | Precision > speed for semantic rules |
| Fallback | panic if `--with-llm` and `claude` not found | Explicit failure, no silent skip |
| Without flag | Zero behavior change | `comply src/` works exactly as before |

## CLI

```
comply --with-llm src/          # Enable LLM rules
comply --with-llm --model haiku # Override model (default: sonnet)
comply --llm-concurrency 4      # Override max parallel subprocesses (default: 8)
```

## Cache Schema

```sql
CREATE TABLE IF NOT EXISTS llm_cache (
    cache_key TEXT PRIMARY KEY,
    diagnostics_json TEXT NOT NULL,
    created_at INTEGER NOT NULL
);
```

No TTL. Invalidation by hash change (content or rule_version).

## LLM Rule: `llm-comment-quality`

### Prompt (comprehensive, from coding-standards skill)

The prompt sends the comment + the code it documents, then asks the LLM to evaluate against ALL criteria:

**Why & consequences:**
- Does the comment answer "what goes wrong if I delete this?"
- Does it chain cause → effect?
- Is inaction justified (empty branch, no-op, early return)?

**Concepts & domain:**
- Are technical terms explained on first use? (what it is, what it does, why it exists)
- Plain language first, jargon in parentheses?
- Structs/types: does it describe the ROLE, not the fields?
- State transitions: past tense for what happened, present for conclusion?
- Limits/invariants/boundaries: WHY the cap + what happens to leftovers?

**Tone & style:**
- Conversational, not mechanical?
- Concrete (specific numbers, names, thresholds) over abstract?
- Complete sentences, active voice, capitalized?
- Emphatic word at end (WHY lands last)?

**Anti-patterns to flag:**
- Paraphrase of code (`// handle click` on `handleClick`)
- Label instead of prose (`// cache expiry 30m`)
- Passive voice (`// Failed jobs are retried`)
- Abstract without specific (`// Retry with backoff` — what count? what interval?)
- Fields listed instead of role (`// Contains failure_percent, last_status`)

### Response Schema

```json
{
  "type": "object",
  "properties": {
    "verdict": { "type": "string", "enum": ["good", "needs_improvement"] },
    "issues": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "criterion": { "type": "string", "enum": [
            "paraphrases_code", "no_consequence", "abstract_not_concrete",
            "passive_voice", "describes_fields_not_role", "no_domain_explanation",
            "mechanical_tone", "wrong_emphasis_order", "missing_cause_effect",
            "label_not_prose", "missing_return_contract", "missing_gotcha_warning"
          ]},
          "explanation": { "type": "string" }
        },
        "required": ["criterion", "explanation"]
      }
    },
    "rewrite_suggestion": { "type": "string" }
  },
  "required": ["verdict", "issues"]
}
```

## Other LLM Rules

### `llm-function-abstraction-levels`
- **Input:** function body (>20 lines)
- **Prompt:** "Does this function mix high-level orchestration with low-level details (regex, byte manipulation, string parsing)?"
- **Schema:** `{ verdict: "clean" | "mixed", high_level_calls: [...], low_level_details: [...] }`

### `llm-intent-naming`
- **Input:** exported function name + its body
- **Prompt:** "Does the name describe INTENT (closeAccount) or IMPLEMENTATION (setStatusToClosed)?"
- **Schema:** `{ verdict: "intent" | "implementation", suggestion: "..." }`

### `llm-pii-in-logs`
- **Input:** logger/console call + surrounding context
- **Prompt:** "Does this log statement contain PII (email, SSN, token, password, full name, phone, address)?"
- **Schema:** `{ verdict: "safe" | "pii", fields: ["email", ...] }`
